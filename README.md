## 📋 О проекте

### Контекст и проблема

Формирование эффективных команд — критическая задача для любого бизнеса. Неправильно подобранная команда приводит к:
- **Конфликтам** между участниками из-за несовместимости психотипов
- **Низкой продуктивности** из-за дисбаланса ролей и мотиваций
- **Текучести кадров** — люди уходят из некомфортных коллективов
- **Провалу проектов** — даже сильные специалисты не могут работать вместе

### Решение

MISIS Bites — это интеллектуальный сервис подбора команд, который использует:
1. **DISC-профилирование** — оценка поведенческих типов (Dominance, Influence, Steadiness, Compliance)
2. **Методику Герчикова** — анализ типов мотивации (Инструментальный, Профессиональный, Патриотический, Хозяин, Избегающий)
3. **Алгоритм совместимости** — математический расчёт совместимости участников
4. **LLM-рекомендации** — генерация развёрнутых рекомендаций по каждому кандидату

## 🧮 Алгоритм расчёта совместимости

### Формула совместимости

Общая совместимость рассчитывается как взвешенная сумма двух компонентов:

```
S_total = w₁ × S_disc + w₂ × S_gerchikov
```

где:
- `S_disc` — совместимость по DISC (нормализованные значения 0-1)
- `S_gerchikov` — совместимость по Герчикову (нормализованные значения 0-1)
- `w₁, w₂` — весовые коэффициенты (настраиваются под требования кейса)

### Расчёт DISC-совместимости

**Ключевая идея:** кандидат должен усиливать слабое звено команды, а не дублировать сильные стороны.

Алгоритм расчёта:

1. **Находим слабую ось команды** — определяем минимальное значение среди средних D, I, S, C по команде

2. **Рассчитываем три компонента:**

   - **L (Lift)** — насколько кандидат поднял минимум команды:
     ```
     L = max(0, новый_минимум − старый_минимум) / макс_возможный_подъём
     ```
   
   - **W (Weakness contribution)** — сила кандидата на слабейшей оси команды:
     ```
     W = значение кандидата на слабейшей оси (0-1)
     ```
   
   - **P (Penalty)** — штраф за создание новой дыры в другой оси:
     ```
     P = max(0, старый_минимум − новый_минимум_в_другой_оси)
     ```

3. **Итоговый DISC-скор:**
   ```
   S_disc = 0.50 × L + 0.35 × W + 0.15 × (1 − P)
   ```

**Пример:** Если у команды проседает S (стабильность), а кандидат имеет высокий S — он закроет эту слабость и получит высокий скор. Если же кандидат — чистый лидер с низким S, он не усилит команду и получит низкий скор.

### Расчёт совместимости по Герчикову

**Ключевая идея:** кандидат должен разделять мотивацию команды для минимизации культурных конфликтов.

Используется **косинусное сходство** между мотивационным профилем кандидата и средним профилем команды:

```
S_gerch = cos(g_mean, g_c) = (g_mean · g_c) / (|g_mean| × |g_c|)
```

где:
- `g_mean` — средний вектор мотивации команды [IN, PR, PA, HO, LU]
- `g_c` — вектор мотивации кандидата

**Значения:**
- **1.0** — идеальное совпадение мотиваций
- **0.7-0.9** — хорошее совпадение
- **< 0.7** — риск мотивационного конфликта

### Соответствие роли (S_role)

Проверка соответствия профиля кандидата эталонному профилю для конкретной роли:

```
S_role = 0.5 × cos(d_c, d*) + 0.5 × cos(g_c, g*)
```

где `d*` и `g*` — эталонные профили DISC и Герчикова для данной роли.

**Примеры эталонных профилей:**
- **Аналитик**: высокий C (точность), PR (профессиональный)
- **Продажник**: высокий I+D, IN (инструментальный)
- **Тимлид**: сбалансированный D+I, HO (хозяйский)

### Итоговая формула

```
S = 0.40 × S_disc + 0.35 × S_gerch + 0.25 × S_role
```

- **40%** — DISC-совместимость (усиление слабого звена)
- **35%** — Мотивационная совместимость (разделяет ли ценности команды)
- **25%** — Соответствие роли (профессиональная пригодность)

## 🏗 Архитектура бэкенда

### Мультимодульная структура

```
backend/
├── api/              # Основной модуль с REST API, сервисами и сущностями
├── llm-api/          # Модуль интеграции с LLM для генерации рекомендаций
├── compute/          # Вычислительный модуль с алгоритмами совместимости
└── build.gradle      # Корневой файл сборки
```

### Модуль `compute`

Выделен в отдельный модуль для:
- **Изоляции бизнес-логики** — алгоритмы не зависят от Spring
- **Переиспользования** — можно использовать в других проектах
- **Тестируемости** — юнит-тесты без контекста Spring
- **Масштабируемости** — при необходимости можно вынести в микросервис

**Ключевые классы:**
- `CompatibilityCalculator` — основной калькулятор совместимости
- `TeamMemberMetrics` — интерфейс для доступа к метрикам участника

### Модуль `api`

Содержит:
- **Контроллеры** — REST endpoints (`/teams`, `/candidates`)
- **Сервисы** — бизнес-логика (`TeamService`, `RecommendationService`)
- **Репозитории** — доступ к данным (Spring Data JPA)
- **DTO** — объекты передачи данных
- **Entity** — JPA-сущности

### Модуль `llm-api`

Интеграция с LLM для генерации рекомендаций:
- Отправка профилей кандидатов в LLM
- Получение структурированных рекомендаций (pros, cons, recommendation)

## 🤖 Как работают рекомендации

### 1. Сбор данных

При запросе рекомендаций система собирает:
- Профиль кандидата (DISC + Герчиков)
- Профили всех участников команды
- Средние показатели по команде
- Требуемую роль

### 2. Расчёт совместимости

Вычислительный модуль рассчитывает:
- Индивидуальную совместимость с каждым участником
- Среднюю совместимость с командой
- Вклад в общую динамику команды

### 3. Генерация LLM-рекомендаций

Данные передаются в LLM через `llm-api` модуль:
```json
{
  "type": "CANDIDATE",
  "teamAnalytics": { "avgDISC": {...}, "avgGerchikov": {...} },
  "candidate": { "discD": 0.85, "discI": 0.32, ... },
  "teamMembers": [...]
}
```

### 4. Формирование ответа

LLM возвращает структурированный ответ:
- **Pros** — сильные стороны кандидата для этой команды
- **Cons** — потенциальные проблемы/зоны роста
- **Recommendation** — итоговая рекомендация с обоснованием

## 🚀 Развёртывание (Deployment)

### Требования
- Docker 20+
- Docker Compose 2+
- Java 17+ (для локальной разработки)
- Node.js 18+ (для локальной разработки фронтенда)

### Быстрый старт через Docker Compose

```bash
# Клонировать репозиторий
git clone <repository-url>
cd tulahack2026-misis-bites

# Запустить все сервисы
docker-compose up --build

# Остановить сервисы
docker-compose down
```

### Сервисы

| Сервис | Порт | Описание |
|--------|------|----------|
| Frontend (Nginx) | 80 | Vue.js приложение |
| Backend (Spring Boot) | 8080 | REST API |
| PostgreSQL | 5432 | База данных |

### Локальная разработка

```bash
cd backend

# Запустить приложение
./gradlew bootRun

./gradlew test
```
